AutoPR: Building an Autonomous Agent That Finds Bounties and Ships Pull Requests

June 2026 · source on GitHub · built for the Qwen Cloud Global AI Hackathon (Autopilot Agent track)

Open-source bounty hunting has a brutal property: it is a speed game. When a $250 issue opens on a popular bounty repo, the first credible proposal usually appears within minutes. I once watched a one-hour-old issue accumulate eighteen competing proposals. A human checking email a few times a day is structurally unable to compete — not for lack of skill, but for lack of latency.

So I built AutoPR: an agent that runs the entire loop unattended — scan bounty platforms, decide whether an issue is worth attempting, clone the repo, write the fix with real coding tools, run the tests, and open the pull request. This post covers the engineering decisions that mattered, including the ones I got wrong first.

Five components, one process, one SQLite file. The dashboard streams every event over server-sent events, so I can watch the agent think in real time from a browser tab.

Two models, deliberately

AutoPR uses Qwen-Max for triage and Qwen-Plus for coding, and the split is the single best architectural decision in the project. Triage is a judgment problem: read an issue, estimate whether a fix is achievable by an agent, output a confidence score and a technical approach. That benefits from the strongest reasoning model available. Coding, by contrast, is dozens of tool-loop iterations per attempt — read a file, patch it, run the tests, read the failure, patch again. Paying frontier-reasoning prices per iteration would make the economics absurd.

StageModelWhy
TriageQwen-MaxOne expensive, high-stakes judgment per issue
CodingQwen-PlusMany cheap, fast iterations per attempt

Triage returns structured JSON — a score from 0 to 1, a skip flag, and a proposed approach. The approach field matters more than the score: it becomes the opening context for the coder, so the expensive model effectively writes the cheap model's briefing.

Six tools are enough

The coder is a classic tool-loop agent. My first instinct was to give it a rich toolkit — git operations, dependency installation, web search, linters. Almost all of it was a mistake. Every added tool widens the space of wrong actions, and the model spends turns exploring tools instead of fixing the bug. The shipped version has exactly six:

  • list_files and read_file — orientation
  • search_code — find the offending symbol fast instead of reading the whole tree
  • write_file — the actual fix
  • run_command — run the test suite, nothing fancier
  • finish() — the agent's explicit declaration that it is done, with a summary

finish() deserves a note: making termination an explicit tool call, rather than inferring it from the model's prose, eliminated a whole class of half-finished attempts. If the agent never calls finish() within the iteration budget, the attempt is marked failed and nothing is submitted. No PR is better than a broken PR.

The defensive layer is the real product

The naive version of this agent loses money. The profitable version is mostly guardrails:

  • Competing-PR check before spending a single token. If the issue already has linked PRs or a swarm of proposals, the expected value is near zero and the scanner skips it. This one check eliminates the majority of wasted attempts.
  • Per-repo merge-rate memory. Every attempt's outcome lands in SQLite. After five attempts on a repo, AutoPR computes a merge rate; below 20%, the repo is blacklisted. Some maintainers never merge outside contributions — the agent learns who they are so I stop paying to find out.
  • Test-gated submission. If run_command can find a test suite and it fails after the patch, the attempt dies locally. The PR submitter only ever sees green attempts.

The uncomfortable lesson: the LLM is maybe a third of the system. The rest is economics enforcement — deciding what not to attempt, remembering what didn't work, and refusing to ship anything unverified.

Watching it work: SSE over WebSockets

The dashboard is FastAPI serving a single page that subscribes to an event stream. I chose server-sent events over WebSockets because the data flow is strictly one-directional — the browser never talks back — and SSE gives automatic reconnection for free. Every component emits typed events (scanned, triaging, coding, pr_open), which also makes the demo video pipeline trivial: replay a recorded event log through the real dashboard and screen-record it.

Costs and honest numbers

The whole thing runs on a small Alibaba Cloud ECS instance (~$8/month) with pay-per-token model calls through Model Studio. A full scan-triage-code-submit cycle costs cents; triage-and-skip costs a fraction of a cent. The economics work precisely because the defensive layer ensures the expensive path only runs on issues with a real chance of paying out.

Is it a money printer? No. Bounty markets are competitive, and merge decisions belong to human maintainers with their own preferences and timelines. What the agent buys is latency and coverage: it is always watching, it never gets bored triaging, and it only escalates to expensive work when the numbers justify it. That pattern — cheap continuous monitoring, LLM judgment at the gate, expensive action behind guardrails — generalizes far beyond bounties; it is the same architecture I now use for inbox triage and back-office automation.
